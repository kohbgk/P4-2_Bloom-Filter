# P4-2 Bloom Filter

Supporting code and data for our INF1008 Data Structures and Algorithms assignment,
*Evaluation of Data Structures or Algorithms using an AI Chatbot* (AY 2025/2026, Trimester 3).

We asked two AI chatbots, **ChatGPT (GPT-5.5)** and **Claude (Opus 4.8)**, the same five
questions about the Bloom filter data structure, then evaluated their answers. The final
question asked each chatbot to generate a working Bloom filter implementation. This
repository holds both generated implementations, the datasets they were tested on, their
measured output, and an interactive visualiser we built to compare their hashing behaviour.

**Live visualiser:** https://kohbgk.github.io/P4-2_Bloom-Filter/Question%206/bloom_animator.html

## Repository structure

```
Question 6/
  bloom_animator.html      interactive visualiser comparing the two implementations
  bloom_urls_claude.py     Bloom filter generated by Claude Opus 4.8
  bloom_urls_gpt.py        Bloom filter generated by ChatGPT 5.5

  non_malicious_urls.txt   999,299 unique benign URLs (queried to measure false positives)

  output_gpt.txt           recorded output of the ChatGPT implementation
```

## Requirements

Python 3, standard library only. No `pip install` is needed, as both implementations were
constrained to the standard library by the question we asked.

## Running the implementations

Run from inside the `Question 6` folder so the scripts can find the two dataset files:

```bash
cd "Question 6"
```

Claude's version takes no arguments and reads the default filenames:

```bash
python bloom_urls_claude.py
```

ChatGPT's version accepts optional arguments:

```bash
python bloom_urls_gpt.py
python bloom_urls_gpt.py --malicious malicious_urls.txt --non-malicious non_malicious_urls.txt --fp-rate 0.01
python bloom_urls_gpt.py --progress-every 0
```

| Argument | Meaning |
| --- | --- |
| `--malicious` | path to the malicious URL file |
| `--non-malicious` | path to the non-malicious URL file |
| `--fp-rate` | target false-positive rate, for example `0.01` for 1% |
| `--progress-every` | print progress every N URLs, `0` disables progress output |

## Results

Both implementations were run on the same datasets at a 1% target false-positive rate.
Full output is recorded in `output_claude.txt` and `output_gpt.txt`.

| Measure | ChatGPT 5.5 | Claude Opus 4.8 |
| --- | --- | --- |
| Malicious URLs inserted | 64,938 | 64,938 |
| Non-malicious URLs queried | 999,299 | 999,299 |
| Dataset overlaps | 0 | 0 |
| False negatives | 0 | 0 |
| False positives | 9,901 | 10,051 |
| Measured false-positive rate | 0.9908% | 1.0058% |
| Theoretical false-positive rate | not reported | 1.0039% |
| Memory used | not reported | 76.0 KiB (9.6 bits/URL) |

Neither implementation produced a false negative, which is the defining guarantee of a
Bloom filter. Both measured rates sit close to the 1% target.

### Why the two differ

Both implementations use the same approach: one SHA-256 digest per URL, split into two
128-bit halves `h1` and `h2`, with probe positions generated as `(h1 + i * h2) mod m`.
The difference is in how `h2` is treated. ChatGPT uses the raw `h2` value unless it is
zero, whereas Claude applies `h2 | 1` to force it odd. For roughly half of the URLs this
changes the step value, producing a different pattern of set bits and therefore a
different set of false positives.

Forcing `h2` odd is not automatically more correct here. The filter uses m = 622,435 bits,
which is not a power of two, so oddness alone does not guarantee that `h2` is coprime with
m or that the probe sequence covers every position. Both are valid double-hashing
implementations that simply produce different probe sequences.

## Datasets

`malicious_urls.txt` and `non_malicious_urls.txt` each contain one cleaned, unique URL per
line, with no overlap between the two files.
